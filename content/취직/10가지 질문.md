### 1. 최근에 스스로 학습하고 개발한 경험이 있다면 자유롭게 서술해 주세요.
TexTube 프로젝트는 웹, 나아가 iOS 앱까지 목표로 한 프로젝트입니다. 유튜브 영상 URL을 AI를 이용해 유저가 글로 옮기는 파이프라인을 '작성'이라고 명명하여, 작성한 게시글로 이루어진 플랫폼을 설계, 개발했습니다. 웹 백엔드/프론트엔드 프레임워크(NestJS, Next.js)를 12월부터 직접 공부하기 시작해서 1월 8일 프로토타입 기능을 구현하였습니다.

### 2. 지금까지 살면서 달성했던 목표 중, 가장 뿌듯했던 경험에 대해 서술해 주세요.
일 또는 공부에 재미를 느끼고, 그 결과값이 인정받는 순간에 뿌듯함을 느낍니다. 면접과는 동떨어진 이야기지만, 고등학교 2학년 동안 모의고사 성적이 모든 과목을 5등급씩 올렸던 (국영수 5,6,7 → 1,2,3) 기억이 가장 인상깊게 남습니다. 공부에 흥미를 느낀 것이 성적으로 증명되었기에 그랬던 것 같습니다. 개발에 있어서는 제가 만든 것이 직접적으로 수익이나 인정에 영향을 끼치지 않기 때문에 인상적인 기억으로 남지 못했던 것 같습니다. 완전한 앱을 추구하기에, 프로토타입이나 추가로 확장할 거리가 많은 앱에서는 뿌듯함보단 아쉬움을 더 많이 느꼈던 것 같습니다. 현재 취업에 시간을 쏟고 있기 때문에 프로토타입으로 내버려 둔 제 앱에 아쉬움을 많이 느낍니다.

### 3. 스트레스를 해소하는 자신만의 방법이나 루틴이 있다면 소개해 주세요.
사실 친구들과 연락하거나 만나서 허심탄회하게 이야기하는 것이 일반적으로도 저에게도 좋은 방법입니다. 저만의 독특한 방식이라면 믿으실지는 모르겠지만 책을 읽고 글을 쓰는 방식인 것 같습니다. 평소에도 Obsidian이라는 텍스트 에디터를 정말 매일같이 사용하는데, 스트레스 받을 일이 생겼을 때 감정을 글로 훌훌 털어내고는 합니다. 또 이걸로는 도저히 답이 보이지 않을 때엔, 그것을 해결하기 위한 책을 뒤지거나 아예 다른 영역의 호기심으로 도피하고는 합니다. 최근의 예를 들면, 회의감이 드는 관계가 있었는데 \<남자는 왜 친구가 없을까>와 \<우정이란 무엇인가>라는 책을 읽었었고, 그걸로도 답답하면 \<케임브리지 스페인사>나 <1988 서울, 극장도시의 탄생>을 읽는 식의 패턴입니다.

### 4. 일상에 있어서 ‘일’과 ‘삶’ 중 어떤 요소를 더 중요하게 생각하는지, 본인만의 철학이 있다면 자유롭게 적어 주세요.
일은 사회고, 삶은 관계라고 생각합니다. 요즘 삶과 일을 분리해야한다는 풍조가 워라밸이라는 단어로 수렴되고는 합니다. 그것은 현재 사회가 자신의 가치를 느끼기 어렵도록 되어버렸기 때문이라고 생각합니다. 각종 생활비는 점점 증가되는 한편 아이와의 관계마저 '일'로 치환되기에 이르렀습니다. 아이마저 '투자'의 대상이 되고 그만큼의 일을 해내야만 하니까요. 그 '일'이 하기 싫은 일이라면 고통이 몇배로 늘어납니다.
저는 결국 '일'이라는 것에 다시금 가치를 부여해야 한다고 생각합니다. 워라밸이 아닌 자신에게 진정으로 의미가 있는 바에 힘써 사회적 가치를 창출해낸다면, 그것이 결국 '삶'을 위한 동력이 될 수 있다고 생각합니다. '일'이 나에게 주요 동력이 된다면, 저에겐 저녁과 주말의 '삶'이 있습니다. 저녁과 주말에도 정말 급하고 필요한 일이라면 무릅쓰고 일하더라도, 저에게 주어진 시간이 부족하다고 생각하지 않습니다.

### 5. 최근에 사용한 기술 스택 중, 헬스케어 서비스에 특히 적합하다고 생각한 기술이 있다면 설명해 주세요.
헬스케어는 민감한 개인 정보를 다루므로 보안과 안정성이 중요합니다. Next.js의 SSR(Server-Side Rendering) 방식은 민감한 로직을 서버에서 처리해 클라이언트 노출을 최소화할 수 있고, React의 컴포넌트 기반 구조는 의료 데이터 표시 같은 복잡한 UI를 모듈화하여 유지보수와 테스트를 용이하게 합니다.

### 6. 실 서비스에서 발생한 프론트엔드 성능 저하 또는 상태 관리 문제를 하나 설명하고, 원인 분석 부터 해결까지의 과정을 구체적으로 작성해주세요
아직 실 서비스 운영 경험이 없어 직접 겪은 사례는 말씀드릴 수 없습니다. 대신 프론트엔드 성능 최적화를 따로 공부하면서 인상 깊었던 사례가 있었는데, 우아한형제들 기술 블로그에서 본 배민 메인 페이지 성능 개선 케이스였습니다. 해당 케이스에서는 TTI가 5초 이상 걸리는 문제가 있었습니다. 분석해보니 모든 라우트의 코드가 하나의 번들로 묶여서 초기 로딩이 무거웠던 게 원인이었고, React.lazy와 Suspense로 Route-based Code Splitting을 적용해서 TTI를 2초로 줄이고 번들 크기도 40% 감소시켰다고 합니다.
이 글을 읽으면서 성능 문제는 감으로 접근하는 게 아니라 Lighthouse나 Web Vitals 같은 도구로 제대로 측정해야 한다는 걸 배웠습니다. 그리고 개선할 때도 사용자가 실제로 체감하는 지표를 우선순위로 잡아야 효과적이라는 점도 인상 깊었습니다. 실무 경험은 아직 없지만 이런 식으로 실제 사례들을 공부하면서 문제 해결 방식을 익히려고 노력했고, 입사하면 실제 서비스에서 실제 성능 이슈를 직접 경험하고 해결하면서 성장하고 싶습니다.

### 7.  포트폴리오에서 진행 해주신 프로젝트에서 상태 관리를 어떻게 설계했고, 성능 이슈를 어떻게 해결 했는지 구체적인 사례를 설명해주세요. 
TexTube 프로젝트는 현재 MVP 단계로 복잡한 상태 관리나 성능 이슈를 경험하지 못했습니다. 현재 상태 관리 구조는 UI 상태를 useState로 관리하고, 서버 데이터는 Next.js Server Component에서 처리하며, 폼 상태는 useState로 관리합니다. Redux 등의 전역 상태 관리 라이브러리는 사용하지 않았는데, 컴포넌트 구조가 단순해 props를 2-3단계만 전달하면 되고, 전역으로 공유할 복잡한 상태가 없기 때문입니다.
성능 문제를 직접 겪지는 않았지만, 향후 확장을 고려한 설계를 했습니다. AI 텍스트 생성처럼 긴 처리 시간이 필요한 경우 Suspense와 Error Boundary로 로딩과 에러 상태를 선언적으로 처리했습니다. Next.js 학습을 통해 Server Component와 Client Component 분리, React Query를 활용한 서버 상태 관리 등을 배웠고, 서비스 규모가 커지면 적용할 계획입니다. 솔직히 아직 실제 성능 문제를 측정하고 해결한 경험은 부족하지만, 문제를 어떻게 인식하고 접근해야 하는지는 학습하며 준비하고 있습니다.

### 8. 개발자로서 협업 또는 실무에서 가장 중요하게 생각하는 점은 무엇인가요?
협업에 있어 가장 중요한 것은 실무 조직원이 빠르게 이해하고 수정할 수 있는 개발 방식입니다. 그래야만 회의를 통해 곧바로 의사결정을 적용할 수 있다고 생각합니다. 
제 프로젝트에서는 세세한 모듈화와 코드 가독성을 가장 중요하게 생각하고 있습니다. 그리고 각 모듈을 즉석으로 테스트할 수 있는 페이지를 따로 만들어 다른 사람이 제 프로젝트에 참여할 경우 즉석으로 이해할 수 있도록 계획했습니다. 이 테스트는 Mock API 등을 구축하여 실제 데이터가 없어도 가능하도록 합니다. 즉 소속 개발자가 즉석으로 이해할 수 있는 페이지입니다.
또한 프로토타입이 완성된 후 개발자가 아닌 다른 팀원이 비즈니스 로직을 이해할 수 있는 간단하고 구조화된 문서 페이지를 만드려고 계획했습니다. Git 버전 관리를 혼자서도 다른 개발자가 참여할 수 있는 방식으로 진행하며, 로그를 통해 빠르게 버그를 처리하는 것도 가장 중요한 요소 중 하나입니다.
서비스 측면에서는 결국 속도, 웹 사이트 몰입을 위한 일관적인 디자인과 상호 작용의 연출이 가장 중요합니다. 디자이너가 따로 없을 경우 전체적인 시스템을 유저 경험을 최우선화하는 방식으로 설계할 것입니다. 디자이너가 있다면 유저가 웹 사이트와 더욱 상호작용을 이끌 수 있도록 세부적인 UI에 집중할 것입니다.

### 9. 개발 업무 외에, 팀이나 조직에 기여하거나 주도적으로 이끌었던 경험이 있다면 자유롭게 서술해 주세요.
Ufriend 프로젝트는 조별 과제에서 시작했습니다. 제 여러 경험 -- 교내 프로그램 참여 등으로 외국인 유학생들과 많은 시간을 보내고 여러 고충을 들었습니다 -- 에 기반한 직관으로 아직 수익이 창출되지 않은 서비스 영역을 해결할 아이디어를 제공하고, 주기적으로 회의를 열어 팀원에게 구체화하는 몰입 경험을 제공하였습니다. 그 동기로 실제 수요조사--교내 외국인 유학생들의 100명의 설문 조사 실시--를 진행하고 공모전 참여까지 이끌어냈습니다. 비록 그때 시간이 짧아 입상하지는 못했고, 저는 졸업하여 그만두게 되었지만, 그곳에 남은 팀원은 따로 창업 동아리를 만들어 그 프로젝트를 이어나가 이익 창출을 도모하고 있습니다.
