# Webhook
## 개요
웹 기술: 서버 간에 자동으로 실시간 데이터를 전달

이벤트 기반: 
이벤트 발생 ⇒ URL로 데이터 전송

API와의 차이:
요청 ⇒ 데이터 가져옴

## 특징
### 1. 이벤트 기반
**서버**에서 발생한 특정 이벤트를 감지하고 지정된 엔드포인트로 데이터 전송
- 예: 파일 업로드, 결제 완료
- 사례: GitHub 내의 Pull Request 발생 ⇒ Webhook URL로 데이터 전달

### 2. 실시간 데이터 전송
요청 없이 클라이언트에게 데이터 전송
- 예: 알림, 업데이트
- 사례: 결제 시스템에서 성공 ⇒ Webhook 트리거 ⇒ 성공 메시지 전송(외부)

### 3. 경량화
HTTP 프로토콜 사용
POST 방식으로 데이터 전송
필요한 정보만 전송

## 작동 원리
서버 내에서
1. 서비스: Webhook URL 설정, 데이터 전송 대상(목표) 서버의 엔드포인트
2. 트리거: 이벤트 발생 ⇒ Webhook 트리거
3. 방식: HTTP POST, 데이터 형식: JSON
4. 수신: Webhook이 성공적으로 처리되었음을 응답으로 반환. 200 OK

## 사용 사례
### 결제 서비스
결제 서버에서 결제가 성공/실패했을 때, 사용자의 시스템에 실시간으로 전달
- From: Stripe, PayPal
- To: Client

### CI/CD 파이프라인
코드 변경 시, CI/CD 서버에 이벤트 전송, 자동 빌드 및 테스트
- From: GitHub, GitLab
- To: Jenkins, Travis CI

### 알림 시스템
고객 지원 도구에서 새로운 티켓이 생성되면, 알림을 보낼 수 있음
- From: Zendesk
- To: Email, Slack

### 마케팅 자동화
이메일 마케팅 툴에서 구독자가 추가/삭제시, 자동으로 데이터 전송, 동기화
- From: MailChimp
- To: CRM System

## 장단점
### VS
- **API**: 호출 ⇒ 응답, 실시간 통신 시 폴링 방식
- **Webhook**: 이벤트 ⇒ 호출 ⇒ 응답, 실시간 또는 이벤트 통신

### 장점
- 효율성: 네트워크 리소스 절약
- 실시간: 변경 사항 및 중요한 이벤트 발생 시 즉시 반응
- 간단함: 설정 및 구현 간단, RESTful API 사용

### 단점
- [ ] 보안: 공개된 URL 사용
	- 인증 및 서명 방식으로 보안 강화
- 신뢰성: 네트워크 오류 및 서버 오류 발생시, 이벤트 전달 실패 가능성
	- 재시도 로직 필요
