### vs [[Spring Boot]]
장점
- 생태계: Spring Data, Spring Security, [[Spring Batch]]
	- 엔터프라이즈 요구사항에 대한 솔루션 이미 존재
- 타입 안정성, 컴파일 타임 검증: Java/Kotlin 정적 타이핑 + Spring의 Annotation 기반 검증
	- 런타임 오류를 컴파일 단계에서 많이 잡음
- 성숙도와 안정성: 20년 가까운 역사, 대규모 트래픽/복잡한 도메인에서 검증된 패턴
	- DDD, Hexagonal 아키텍처와 잘 맞음
- 한국에서의 점유율: 금융, 대기업, SI에서 표준
- JVM 성능: 워밍업 이후 처리량과 메모리 관리가 Node 런타임보다 대체로 우수
	- 특히 CPU-bound 작업

단점
- **러닝커브와 보일러플레이트**: 어노테이션 마법(`@Autowired`, `@Transactional` 등)이 내부 동작을 가리는 경우가 많아서, 디버깅 시 "왜 이게 동작하지 않는가"를 추적하기 어려움
- **느린 시작 시간/개발 피드백 루프**: 빌드-재시작 사이클이 Node 대비 느림 (DevTools, JRebel 같은 도구로 완화 가능하지만 근본적 한계)
- **JSON/JS 생태계와의 거리**: 프론트엔드(특히 Next.js)와 같은 언어로 통일하고 싶다면 컨텍스트 스위칭 비용 발생

### NestJS
장점
- **개발 속도**: Prisma, Railway 같은 모던 툴체인과 통합이 빠르고, 1인 개발/소규모 팀에서 프로토타이핑 속도가 빠름
- **TypeScript 단일 언어**: Next.js 프론트와 타입을 공유하거나 적어도 사고방식을 통일할 수 있음. 너의 Note OS적 "통합된 시스템" 철학과도 맞닿아 있음
- **Spring 스타일 구조를 JS 생태계에 가져옴**: 모듈, 데코레이터, DI 컨테이너 — Express의 자유도와 Spring의 구조 사이 중간 지점
- **유연한 어댑터**: 기본은 Express지만 Fastify로 스왑 가능, GraphQL/WebSocket/마이크로서비스 지원이 동일 프레임워크 안에서 일관됨

단점
- **런타임 타입 안정성의 한계**: TypeScript는 컴파일 타임에만 타입을 보장하므로, 외부 입력 검증은 결국 class-validator 등 별도 레이어에 의존해야 함
- **생태계가 Spring보다 얕음**: 엔터프라이즈급 패턴(분산 트랜잭션, 배치 처리 등)에서는 직접 구현하거나 서드파티에 의존해야 하는 경우가 많음
- ==**Node 단일 스레드 모델의 한계**==: CPU-heavy 작업(예: 영상 처리, 대량 데이터 변환)에서는 워커 스레드를 직접 관리해야 함 — TexTube의 YouTube 처리 파이프라인 같은 작업에서 체감했을 수도
- **한국 채용시장에서는 상대적으로 좁음**: 스타트업 중심, 대기업/금융권에서는 드묾
