## 언제 찍을 것인가 (When)

**찍어야 하는 것:**

- 시스템 경계를 넘는 순간 — HTTP 요청/응답, 외부 API 호출, DB 쿼리
- 상태가 바뀌는 순간 — 생성, 수정, 삭제, 트랜지션
- 실패와 예외 — 모든 에러, 특히 예상치 못한 것
- 보안 이벤트 — 인증 시도, 권한 변경

**찍지 않아야 하는 것:**

- 비밀번호, 토큰, 개인정보 (PII)
- 루프 내부의 반복적인 노이즈
- 이미 다른 곳에서 충분히 기록되는 것

---

## 무엇을 찍을 것인가 (What)

로그 한 줄에는 맥락이 있어야 한다. 최소한:

```
timestamp | level | service | trace_id | message | context
```

**trace_id / correlation_id**가 핵심. 하나의 요청이 여러 서비스를 거칠 때 흐름을 추적할 수 있는 유일한 방법.

---

## 어떤 수준으로 찍을 것인가 (Level)

|Level|용도|
|---|---|
|DEBUG|개발 중 상세 추적. 프로덕션엔 꺼둔다|
|INFO|정상 흐름의 주요 이벤트|
|WARN|문제는 아니지만 주의가 필요한 상황|
|ERROR|처리된 실패. 즉각 대응 필요하진 않음|
|FATAL|시스템이 계속 동작할 수 없는 수준|

레벨을 잘못 쓰는 패턴 — 모든 걸 ERROR로 찍거나, 실제 에러를 INFO로 묻어버리는 것.

## 구조화 vs 자유 텍스트

자유 텍스트 로그는 읽기 쉽지만 쿼리하기 어렵다. **구조화 로깅(structured logging)** — JSON 형태 — 이 현대적 표준.

```json
{
  "timestamp": "2026-03-20T10:00:00Z",
  "level": "ERROR",
  "service": "textube-api",
  "trace_id": "abc-123",
  "message": "YouTube transcript fetch failed",
  "video_id": "dQw4w9WgXcQ",
  "error": "429 Too Many Requests"
}
```

사람이 읽을 수 있으면서, 동시에 기계가 집계할 수 있다.

## 핵심 철학

**로그는 미래의 자신에게 보내는 편지다.** 장애 상황에서, 잠을 못 잔 상태로, 압박 속에서 읽힌다는 전제로 써야 한다.

좋은 로그의 기준 하나 — _"이 로그만 보고 무슨 일이 있었는지 재구성할 수 있는가?"_
